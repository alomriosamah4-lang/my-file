# Android CI and Signing (GitHub Actions)

This document describes the CI workflow and how to produce signed APKs via GitHub Actions.

## What the workflow does
- Runs on `push` and `pull_request` to `main`/`master`.
- Sets up Java 17 and installs Android command-line tools (platforms, build-tools, NDK, CMake).
- Builds `assembleDebug` and `assembleRelease` using the project's Gradle wrapper.
- Uploads the built APKs as workflow artifacts.

## Signing the Release APK
The workflow can sign the release APK automatically if you provide the following GitHub repository secrets:
- `KEYSTORE_BASE64` — The keystore file encoded as base64.
- `KEYSTORE_PASSWORD` — Keystore password.
- `KEY_ALIAS` — Key alias in the keystore.
- `KEY_PASSWORD` — Key password.

If `KEYSTORE_BASE64` is present, the workflow will:
1. Decode it into `android/keystore.jks`.
2. Write `android/keystore.properties` with the provided passwords and alias.
3. Gradle will use `keystore.properties` to sign the release APK.

> If secrets are not present, the release APK will be produced unsigned and still uploaded as an artifact.

## Local build (developer)
1. Make sure you have the Android SDK and NDK installed.
2. From repository root:

```bash
cd android
./gradlew clean assembleDebug assembleRelease
```

To build a signed release locally, create `keystore.properties` in the repository root (`android/keystore.properties`) with:

```
storeFile=keystore.jks
storePassword=YOUR_STORE_PASSWORD
keyAlias=YOUR_ALIAS
keyPassword=YOUR_KEY_PASSWORD
```

Place your `keystore.jks` file at `android/keystore.jks`.

## Notes & Best Practices
- Keep keystore and passwords in GitHub Secrets — never commit them to the repo.
- Use versioned `build-tools`, `ndk`, and `cmake` in the workflow to ensure reproducible builds.
- Consider enabling a signing flow that uses Google Play App Signing for production releases.
